# Privacy

Clickit records what you copy. That makes it, by design, one of the most sensitive applications on your machine — clipboard history accumulates passwords, API tokens, private messages, addresses and screenshots without you ever deciding to save them.

This document states exactly what Clickit does with that data.

## The short version

Clickit does not send your data anywhere. There is no server. There is no account. There is no network code in the application at all.

## What Clickit collects

Only what you copy, and only while monitoring is active:

- Plain text
- URLs
- Images and screenshots that reach the system clipboard

Alongside each item, Clickit records: content type, a SHA-256 hash used for duplicate detection, creation and last-used timestamps, byte size, pinned state, and a best-effort guess at the source application's bundle identifier.

## What Clickit does not do

- **No network requests.** Clickit makes none. Not for updates, not for crash reports, not for anything.
- **No accounts or sign-in.**
- **No cloud storage or sync.** Nothing is uploaded, backed up remotely, or shared between devices.
- **No telemetry or analytics.** Not opt-out; simply absent.
- **No crash reporting to any third party.**
- **No AI, machine learning, or content analysis.** Clickit decides whether content is text, a link, or an image. It does not read it further, does not classify it, and does not run OCR on your screenshots.
- **No advertising or tracking identifiers.**
- **No selling or sharing of data**, because no data ever leaves the machine to sell or share.

## Content other applications ask Clickit not to record

Clickit honours the informal [nspasteboard.org](http://nspasteboard.org) conventions. If an application marks its clipboard write with any of:

- `org.nspasteboard.ConcealedType` — used by password managers for secrets
- `org.nspasteboard.TransientType` — content not meant to be kept
- `org.nspasteboard.AutoGeneratedType` — content not produced by a deliberate user action

then Clickit reads nothing and records nothing. This check happens before classification, so the content never reaches the history at all.

This depends on the source application marking its writes. Not every password manager does, so it reduces exposure rather than eliminating it.

## Where your data lives

Everything is on your Mac, under:

```
~/Library/Application Support/Clickit/
├── clickit.sqlite   History: text, links, and metadata
└── Images/          PNG files for copied images and screenshots
```

Text and metadata are stored in a local SQLite database. It is an ordinary file on your disk, readable only by your user account. It is never uploaded, never synced, and never leaves the machine.

History persists across quitting and relaunching Clickit, but **unpinned history is cleared when the Mac restarts** (on by default). In normal use that is what bounds how long anything is kept.

If the Mac goes a long time without restarting, the retention rules take over: 30 days for text and links, 7 days for images, or sooner if the 1,000-item or 500 MB limits are reached. Pinned items are kept indefinitely and are never removed automatically. Clear History removes records and their image files immediately.

Settings are stored in the standard macOS preferences domain, `com.clickit.Clickit`.

Nothing is written outside the Clickit Application Support directory and that preferences domain.

## Logging

Clickit writes to the unified system log for debugging. **Clipboard contents are never logged, at any level.** Log lines contain metadata only, for example:

```
Captured new text item, 23 bytes
Cleanup removed 3 items (expired: 1, count: 2, size: 0)
```

This restriction exists because the system log is readable by other processes on the machine. Logging clipboard contents would leak everything Clickit is supposed to protect.

## Controls you have

- **Clear history on restart** — on by default, so unpinned history does not outlive the session. Turn it off in Settings if you want history to persist across restarts.
- **Pause Monitoring** — stops the poll timer entirely. Nothing is captured while paused, and resuming does not retroactively pick up what you copied in the meantime.
- **Clear History** — removes everything except pinned items. Settings also offers a destructive action that removes pinned items too.
- **Delete individual items** — removes the record and, for images, the file on disk.
- **Retention limits** — configure how many items, how much disk, and how long each type is kept. Defaults are 1,000 items, 500 MB, 30 days for text and links, and 7 days for images.
- **Excluded applications** — see the limitation below.

## Known limitations

Please read these. They are the difference between what Clickit does today and what a privacy-conscious user might reasonably assume.

**Excluded applications are only partly implemented.** You can list bundle identifiers in Settings and copies attributed to them are dropped before being recorded. However, attribution uses `NSWorkspace.frontmostApplication` at the moment the change is detected. That is a heuristic, not a guarantee of which process wrote to the pasteboard. A background process, or an app that writes while another is frontmost, can be misattributed. **Do not treat exclusions as a security boundary.** More reliable attribution and an app picker are roadmap Phase 4.

**Do-not-record markers only work if the source application sets them.** Clickit honours the `org.nspasteboard` conventions described above, but a password manager that does not mark its writes will have its copies recorded like anything else. When in doubt, pause monitoring before handling credentials.

**History is not encrypted at rest.** The database and image files are readable by anything running as your user. macOS FileVault protects them at the disk level; Clickit adds no further encryption. This matters more now that history persists across restarts than it did when it lived only in memory.

**No sandbox.** Clickit is not sandboxed, because it writes to the conventional `~/Library/Application Support/` location. It requests no special permissions — notably, it does **not** request Accessibility permission, because it never simulates keystrokes.

## Changes to this document

Any change that broadens what Clickit collects or where data goes will be called out in [CHANGELOG.md](CHANGELOG.md) and in the release notes. The commitments in "What Clickit does not do" are architectural, not preferences, and are not intended to change.
