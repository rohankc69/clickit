# Security Policy

## Supported versions

Clickit is in early development and has not had a tagged release. Only the current `main` branch receives fixes.

| Version | Supported |
| --- | --- |
| `main` | Yes |
| Anything else | No |

## Reporting a vulnerability

**Please do not open a public issue for a security or privacy vulnerability.**

Report it privately through GitHub's [private vulnerability reporting](https://docs.github.com/en/code-security/security-advisories/guidance-on-reporting-and-writing-information-about-vulnerabilities/privately-reporting-a-security-vulnerability): go to the repository's **Security** tab and choose **Report a vulnerability**. This creates a private advisory visible only to maintainers.

If you would rather not use GitHub, email **clickitopensource@gmail.com** instead.

Please include:

- What the issue is and why it matters
- Steps to reproduce, or a proof of concept
- Your macOS version and how you built or obtained Clickit
- Any suggested fix, if you have one

**Do not include real clipboard contents, credentials, or screenshots of your clipboard history.** Use synthetic data to demonstrate the problem.

## What to expect

- Acknowledgement within roughly a week
- An assessment of severity and scope, shared with you
- A fix on `main`, with credit in the advisory and [CHANGELOG.md](CHANGELOG.md) unless you prefer otherwise
- Public disclosure once a fix is available

Clickit is a volunteer project without a paid security team, so these are good-faith intentions rather than a contractual commitment.

## Scope

Clickit stores what you copy. The threat model that matters most is **local**: another process, another user on the machine, or a backup reading clipboard history that should not have been retained.

Especially relevant:

- Clipboard contents appearing anywhere they should not — logs, crash reports, temporary files, or any path outside the Clickit Application Support directory
- Any network request originating from Clickit. There should be none, ever. One would be a serious bug.
- Data surviving Clear History, Pause Monitoring, or the retention rules
- Image files left on disk after their records are deleted
- Privilege or permission issues, including anything that would make Clickit request or use Accessibility permission

## Known limitations, already documented

These are recorded in [PRIVACY.md](PRIVACY.md) and are not news. Reporting them is welcome, but they are known:

- **Excluded applications rely on best-effort source attribution** via `NSWorkspace.frontmostApplication` and are not a security boundary.
- **Do-not-record markers depend on the source application.** Clickit honours `org.nspasteboard.ConcealedType`, `TransientType` and `AutoGeneratedType`, but an application that does not set them will have its copies recorded.
- **History is not encrypted at rest.** Image files are plain PNGs, protected only by file permissions and FileVault.
- **History is currently in memory only** and is lost on quit. When persistence lands, at-rest handling will need review.

A report that deepens any of these — a concrete exploitation path, or a workable fix — is genuinely useful.

## Out of scope

- An attacker who already has code execution as your user. Clipboard history is readable by anything running as you; that is a property of the platform.
- Physical access to an unlocked machine.
- Vulnerabilities in macOS itself. Report those to Apple.
- Missing hardening that has no demonstrated impact.
